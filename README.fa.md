# camel

سه تا اسکیل کوچیک برای Claude Code، برای وقتی که یک سشن بیشتر از چیزی که کار واقعی لازم داره توکن می‌خوره: یکی برای خوندن‌های شلخته‌ی فایل، یکی برای زیاده‌روی توی ران‌کردن ایجنت و گرفتن اسکرین‌شات، و یکی که هر دو رو با هم میاره و سخت‌گیرتر هم می‌شه، برای وقتی که خود بودجه تنگه.

## چرا این رو ساختم

یک مدت داشتم می‌دیدم Claude Code سه تا کار رو مدام تکرار می‌کنه، حتی وقتی لازم نبوده: یک فایل کامل رو می‌خونه که فقط دنبال یک تابع بگرده، یک ساب‌ایجنت ران می‌کنه برای کاری که خودش می‌تونست مستقیم انجام بده، و از یک تغییر ظاهری اسکرین‌شات می‌گیره که "چک" کنه، در حالی که فقط کافی بود دیف رو بخونه. هیچ‌کدوم این‌ها به‌تنهایی اشکالی ندارن، ولی توی یک سشن طولانی خیلی زود جمع می‌شن و بیشترشون هم قابل جلوگیری هستن، بدون این‌که چیزی واقعی از دست بره.

برای همین camel یک سری اسکیل با حالت‌های قابل‌سوییچ هست. کامند رو می‌زنی، تا آخر همون سشن این انضباط فعال می‌مونه، و هیچی توی دقتی که کد نوشته می‌شه عوض نمی‌شه.

## توی این ریپو چی هست

| کامند | چیکار می‌کنه |
|---|---|
| `/camel` | انضباط خوندن فایل و صدا زدن ابزارها: قبل از Read با Grep هدفمند پیدا کن، به‌جای خوندن کل فایل با offset/limit فقط همون بخش لازم رو بخون، فایلی که قبلاً خونده شده دوباره خونده نشه، به‌علاوه‌ی فیلتر `cam` برای کوتاه‌کردن خروجی شلوغ کامندها قبل از این‌که وارد context بشه. |
| `/camel stats` | عدد واقعی از همون ترنسکریپت سشن می‌کشه بیرون: تعداد صدا زدن هر ابزار، چندتا از Readها هدفمند بودن و چندتاشون کامل، فایل‌هایی که بیشتر از یک‌بار خونده شدن، مصرف واقعی توکن توی هر turn. هیچی تخمینی نیست. |
| `/camel-dam` | پیش‌فرض ران‌کردن ساب‌ایجنت و گرفتن اسکرین‌شات برای چک‌کردن UI رو خاموش می‌کنه. اگه Claude فکر کنه یک کار واقعاً نیاز به دلگیت‌کردن داره، باید بگه و اول اجازه بگیره. برای چک ظاهری هم می‌گه خودت چی رو چک کن، به‌جای این‌که خودش اسکرین‌شات بگیره. |
| `/camel-pro` | بیشترین کات. هم `/camel` و هم `/camel-dam` رو لود می‌کنه، بعد سخت‌گیرتر می‌شه: یک رجیستر خروجی فشرده و پرچگالی، `cam` به‌صورت پیش‌فرض روی کامندهای شلوغ، برای کارهای معمولی پلن ننویس، به‌جای سوال پرسیدن فرض منطقی بگیر، یک‌جا verify کن. هرچی هم که کات می‌کنه باید بلند بگه. |

هرکدوم یک پوشه اسکیل معمولیه با یک `SKILL.md`. هرکدوم رو که خواستی بریز توی پوشه اسکیل‌هات، کامندها خودشون پیدا می‌شن.

## فیلتر cam

`/camel` یک wrapper کوچیک برای کامندها هم میاره، `camel/scripts/cam.js`، که می‌ره سراغ اون یکی سینک بزرگ توکن: خروجی کامندها. یک `npm install` یا یک ران تست به‌تنهایی می‌تونه چند صد خط بریزه توی context که فقط چندتاش به درد می‌خوره. cam کامند رو ران می‌کنه، کدهای ANSI رو پاک می‌کنه، خط‌های تکراری و نوار پیشرفت رو جمع می‌کنه، نویز معمول install/build رو می‌ندازه دور و لاگ‌های خیلی بلند رو از سر و ته کوتاه می‌کنه، در حالی که هر خطی که شبیه ارور باشه رو نگه می‌داره، حتی اونایی که وسط لاگ گم شدن. با همون exit code خود کامند هم خارج می‌شه، پس هیچی پایین‌دست خراب نمی‌شه.

```bash
node camel/scripts/cam.js -- npm test
# ...خروجی کوتاه‌شده...
# [cam] 464->162 lines, 65% trimmed (exit 0)
```

روی کامندهای ساده کار می‌کنه (پکیج‌منیجرها، تست‌رانرها، بیلدها). برای کامندهایی که خودشون quote یا pipe دارن، مستقیم ران‌شون کن، چون wrapper دوباره از طریق shell ران می‌کنه و quoting ممکنه جابه‌جا بشه. با `--raw` همه خط‌ها می‌مونه ولی بازم ANSI و نویز پیشرفت تمیز می‌شه.

## نصب

مک / لینوکس / WSL / گیت‌بش:
```bash
curl -fsSL https://raw.githubusercontent.com/AmirHosein-Lotfi/Camel/main/install.sh | bash
```

ویندوز (پاورشل):
```powershell
irm https://raw.githubusercontent.com/AmirHosein-Lotfi/Camel/main/install.ps1 | iex
```

این دو اسکریپت فقط ریپو رو دانلود می‌کنن و camel، camel-dam و camel-pro رو می‌ریزن توی `~/.claude/skills/`. اگه پوشه اسکیل‌هات جای دیگه‌ایه، اول `CLAUDE_SKILLS_DIR` رو ست کن. بعدش Claude Code رو ریستارت کن (یا یک سشن جدید باز کن) تا کامندها بیان بالا.

اگه دوست نداری یک اسکریپت رو مستقیم بدی به شل خودت، نصب دستی هم همینجاست:
```bash
git clone https://github.com/AmirHosein-Lotfi/Camel.git
cp -r Camel/camel Camel/camel-dam Camel/camel-pro ~/.claude/skills/
```

## چرا سه کامند جدا، نه یکی

می‌خواستم بشه دقیقاً همون چیزی که توی یک سشن خاص داره توکن می‌خوره رو فعال کرد، نه همه‌چی رو یکجا. سشنی که بیشتر کارش خوندن و گرپ‌کردن کدبیسه فقط به `/camel` نیاز داره. سشنی که پر از تغییرات فرانت‌اِنده و Claude مدام اسکرین‌شات می‌گیره، `/camel-dam` لازمه. سشنی که بودجه‌ش تنگه و کار زیادی جلوشه، `/camel-pro` می‌خواد، که هر دو رو با هم میاره و سخت‌گیرتر هم می‌شه. جداشون نگه داشتم که هیچ‌وقت مجبور نباشی چیزی رو روشن کنی که بهش نیازی نداری.

## این اسکیل چیکار نمی‌کنه

هیچ‌کدوم این اسکیل‌ها باعث نمی‌شن Claude کد بدتری بنویسه، edge caseها رو رد کنه، ریویو سطحی‌تر بزنه، یا بدون چک‌کردن بگه یک چیزی کار می‌کنه. `/camel-pro` تهاجمی‌ترین حالت بین این سه‌تاست، ولی همونم یک قانون ثابت داره: درست بودن کد، امنیت و گفتن صادقانه‌ی هرچی که کات شده، هیچ‌وقت روی میز نیست. اگه چیزی رو کوتاه چک کرده باشه که یک پاس کامل می‌گرفتش، باید بگه.

## از کجا الهام گرفته شده

ایده‌ی یک کامند که رفتار Claude رو برای بقیه‌ی سشن عوض کنه، به‌همراه یک کامند stats که مصرف واقعی رو اندازه می‌گیره نه تخمینی، از استفاده از [caveman](https://github.com/juliusbrussee/caveman) اومد. اون اسکیل هرچی Claude می‌گه رو خیلی کوتاه‌تر می‌کنه و مصرف رو هم همون‌شکلی track می‌کنه. camel همون الگو رو برداشته و گذاشته روی یک مسئله‌ی دیگه: این‌که Claude چجوری فایل می‌خونه و کِی دلگیت می‌کنه، نه این‌که چجوری جواب می‌ده. هیچ کدی از اون پروژه کپی نشده؛ این اشاره برای ایده‌س، نه پیاده‌سازی.
