# 🐫 camel

**سشن‌های Claude Code که با توکن خیلی کمتر، مسیر خیلی بیشتری رو طی می‌کنن.**

چهار اسکیل کوچیک و ترکیب‌پذیر که هدر رفتن توکن رو *قبل از این‌که اتفاق بیفته* می‌گیرن — خوندن شلخته، زیاده‌روی توی ایجنت و اسکرین‌شات، و چک‌کردن چیزهایی که خود کاربر می‌تونه توی یک ثانیه چک کنه.

یک شتر مسیرهای طولانی رو بدون توقف برای آب‌خوردن طی می‌کنه. بیشتر هدررفت توکن توی یک سشن ایجنتیک از متن نمی‌آد — از چیزی می‌آد که وارد context می‌شه: خوندن کامل یک فایل ۲۰۰۰ خطی فقط برای پیدا کردن یک تابع، ران‌کردن یک ساب‌ایجنت برای کاری که می‌شد مستقیم انجامش داد، یک اسکرین‌شات برای "دیدن" یک تغییر UI به‌جای خوندن دیف، یا ران کامل تست‌ها فقط برای اینکه چیزی رو تایید کنی که خود کاربر هم می‌تونست با یک کامند چک کنه. camel یک چک‌لیست ثابته که این هدررفت رو قبل از اجرای ابزار می‌گیره، نه بعدش.

کامند رو می‌زنی، تا آخر سشن این انضباط فعال می‌مونه. **دقتی که کد باهاش نوشته می‌شه هیچ‌وقت عوض نمی‌شه — این فقط سربار ارکستراسیون رو کات می‌کنه، نه دقت کار رو.**

## چرا این رو ساختم

یک مدت داشتم می‌دیدم Claude Code این کارها رو مدام تکرار می‌کنه، حتی وقتی لازم نبوده:

- یک فایل کامل رو می‌خونه که فقط دنبال یک تابع بگرده.
- یک ساب‌ایجنت ران می‌کنه برای کاری که خودش می‌تونست مستقیم انجام بده.
- از یک تغییر ظاهری اسکرین‌شات می‌گیره که "چک" کنه، به‌جای این‌که فقط دیف رو بخونه.
- کل تست‌ها یا بیلد یا دِو سرور رو ران می‌کنه فقط برای تایید چیزی که خود کاربر قرار بود چک کنه.

هیچ‌کدوم این‌ها به‌تنهایی اشکالی ندارن، ولی توی یک سشن طولانی خیلی زود جمع می‌شن و تقریباً همه‌شون هم قابل جلوگیری هستن، بدون این‌که چیزی واقعی از دست بره.

## توی این ریپو چی هست

| کامند | چیکار می‌کنه |
|---|---|
| 🎯 `/camel` | انضباط خوندن فایل و صدا زدن ابزارها: قبل از Read با Grep هدفمند پیدا کن، به‌جای خوندن کل فایل با offset/limit فقط همون بخش لازم رو بخون، فایلی که قبلاً خونده شده دوباره خونده نشه، به‌علاوه‌ی فیلتر `cam` برای کوتاه‌کردن خروجی شلوغ کامندها قبل از این‌که وارد context بشه. |
| 📊 `/camel stats` | عدد واقعی از همون ترنسکریپت سشن می‌کشه بیرون: تعداد صدا زدن هر ابزار، چندتا از Readها هدفمند بودن و چندتاشون کامل، فایل‌هایی که بیشتر از یک‌بار خونده شدن، مصرف واقعی توکن توی هر turn. هیچی تخمینی نیست. |
| 🚧 `/camel-dam` | سه تا پیش‌فرض رو خاموش می‌کنه: ران‌کردن ساب‌ایجنت، گرفتن اسکرین‌شات برای چک‌کردن UI، و ران‌کردن پیش‌دستانه‌ی تست/بیلد/دِو سرور برای چیزی که خود کاربر هم می‌تونست به‌همون راحتی چک کنه. اگه Claude فکر کنه یک کار واقعاً نیاز به دلگیت‌کردن داره، باید بگه و اول اجازه بگیره — و برای بقیه‌ی چک‌ها هم به‌جای مصرف توکن، کامند دقیق و نتیجه‌ی موردانتظار رو می‌گه. |
| ⚡ `/camel-pro` | بیشترین کات. هم `/camel` و هم `/camel-dam` رو لود می‌کنه، بعد سخت‌گیرتر می‌شه: یک رجیستر خروجی فشرده و پرچگالی، `cam` به‌صورت پیش‌فرض روی کامندهای شلوغ، برای کارهای معمولی پلن ننویس، به‌جای سوال پرسیدن فرض منطقی بگیر، یک‌جا verify کن. هرچی هم که کات می‌کنه باید بلند بگه. |

هرکدوم یک پوشه اسکیل معمولیه با یک `SKILL.md`. هرکدوم رو که خواستی بریز توی پوشه اسکیل‌هات، کامندها خودشون پیدا می‌شن.

## فیلتر cam

`/camel` یک wrapper کوچیک برای کامندها هم میاره، `camel/scripts/cam.js`، که می‌ره سراغ اون یکی سینک بزرگ توکن: خروجی کامندها. یک `npm install` یا یک ران تست به‌تنهایی می‌تونه چند صد خط بریزه توی context که فقط چندتاش به درد می‌خوره. cam کامند رو ران می‌کنه، کدهای ANSI رو پاک می‌کنه، خط‌های تکراری و نوار پیشرفت رو جمع می‌کنه، نویز معمول install/build رو می‌ندازه دور و لاگ‌های خیلی بلند رو از سر و ته کوتاه می‌کنه، در حالی که هر خطی که شبیه ارور باشه رو نگه می‌داره، حتی اونایی که وسط لاگ گم شدن. با همون exit code خود کامند هم خارج می‌شه، پس هیچی پایین‌دست خراب نمی‌شه.

```bash
node camel/scripts/cam.js -- npm test
# ...خروجی کوتاه‌شده...
# [cam] 464->162 lines, 65% trimmed (exit 0)
```

روی کامندهای ساده کار می‌کنه (پکیج‌منیجرها، تست‌رانرها، بیلدها). برای کامندهایی که خودشون quote یا pipe دارن، مستقیم ران‌شون کن، چون wrapper دوباره از طریق shell ران می‌کنه و quoting ممکنه جابه‌جا بشه. با `--raw` همه خط‌ها می‌مونه ولی بازم ANSI و نویز پیشرفت تمیز می‌شه.

## چرا چهار کامند جدا، نه یکی

می‌خواستم بشه دقیقاً همون چیزی که توی یک سشن خاص داره توکن می‌خوره رو فعال کرد، نه همه‌چی رو یکجا.

| سشنت این شکلیه... | این رو استفاده کن... |
|---|---|
| بیشتر کارش خوندن و گرپ‌کردن کدبیسه | `/camel` |
| پر از تغییرات فرانت‌اِند و Claude مدام اسکرین‌شات می‌گیره، ایجنت می‌فرسته، یا تست ری‌ران می‌کنه تا "چک" کنه | `/camel-dam` |
| بودجه‌ش تنگه و کار زیادی جلوشه | `/camel-pro` (هر دو رو با هم میاره و سخت‌گیرتر هم می‌شه) |
| رسید می‌خوای | `/camel stats` |

جداشون نگه داشتم که هیچ‌وقت مجبور نباشی چیزی رو روشن کنی که بهش نیازی نداری.

## این اسکیل چیکار نمی‌کنه

هیچ‌کدوم این اسکیل‌ها باعث نمی‌شن Claude کد بدتری بنویسه، edge caseها رو رد کنه، ریویو سطحی‌تر بزنه، یا بدون یک دلیل واقعی بگه یک چیزی کار می‌کنه. `/camel-pro` تهاجمی‌ترین حالت بین این چهارتاست، ولی همونم یک کف ثابت داره: درست بودن کد، امنیت و گفتن صادقانه‌ی هرچی که کات شده، هیچ‌وقت روی میز نیست. اگه چیزی رو کوتاه چک کرده باشه که یک پاس کامل می‌گرفتش، باید بگه.

## از کجا الهام گرفته شده

ایده‌ی یک کامند که رفتار Claude رو برای بقیه‌ی سشن عوض کنه، به‌همراه یک کامند stats که مصرف واقعی رو اندازه می‌گیره نه تخمینی، از استفاده از [caveman](https://github.com/juliusbrussee/caveman) اومد. اون اسکیل هرچی Claude می‌گه رو خیلی کوتاه‌تر می‌کنه و مصرف رو هم همون‌شکلی track می‌کنه. camel همون الگو رو برداشته و گذاشته روی یک مسئله‌ی دیگه: این‌که Claude چجوری فایل می‌خونه، کِی دلگیت می‌کنه، و کِی کار خودش رو چک می‌کنه — نه این‌که چجوری جواب می‌ده. هیچ کدی از اون پروژه کپی نشده؛ این اشاره برای ایده‌س، نه پیاده‌سازی.
