# Agent, screenshot, and self-check dam (camel-dam)

Three specific habits burn tokens disproportionately to the value they add: spawning subagents for tasks that didn't need one, running a render-screenshot-inspect-iterate loop to "see" a UI change instead of just reasoning about the code and the diff, and proactively running a test suite/build/dev server just to confirm something works when the user is sitting right there and could run the same one command themselves in a second. All three are easy defaults to fall into even on simple or moderate tasks, and all three have a real cost — a subagent starts cold and re-derives context already available in the main thread; a screenshot loop adds image tokens and round-trips; a full test/build run dumps output into context that the user could've glanced at for free on their own machine.

`/camel-dam` turns all three off for the rest of the conversation — a camel doesn't make a detour to a watering hole it doesn't need.

## While active

1. **Don't reach for subagents/Workflow as the default.** Handle the task directly in the main thread with your own tools (Read, Grep, Edit, Bash, etc.), including tasks that feel large — size alone isn't a reason to fan out. A subagent should be the exception you ask about, not the default you reach for.
2. **If you genuinely think a task needs a subagent, ask first.** Don't decide on your own that something warrants agents and launch them. State specifically what you think needs delegating and why (e.g. "this needs N independent searches across unrelated areas of the codebase — want me to spawn agents for that, or should I do it myself, serially?") and wait for a yes before spawning anything.
3. **Don't take your own screenshots to verify visual/frontend work.** Skip browser, computer-use, and preview screenshot tools as a way of checking a UI or design change you just made. Instead, tell the user exactly what to check (e.g. "run the dev server and look at the header spacing on mobile, around 600px width") and ask them to send a screenshot back if visual confirmation actually matters.
4. **Don't run checks the user can trivially run themselves.** After a change, don't proactively run the full test suite, a full build, a dev server smoke test, a linter, or a type-check pass purely to confirm "it works" — that's a command the user can run in the same second it'd take you to spend the tokens on it, and they'll see the real output on their own machine anyway. Instead, name the exact command and what a passing result looks like (e.g. "run `npm test` — should show 42 passing" or "run `tsc --noEmit`, should be clean"), and let them run it. Exceptions where running it yourself still makes sense:
   - A test you just wrote or are actively iterating on as part of TDD (the check is the work, not a confirmation step tacked onto finished work).
   - A narrowly-scoped check with no simple user-facing equivalent (e.g. tracing logic by hand, reading the diff, grepping for a regression pattern).
   - The user asked you directly to run something, or working autonomously without the user present (headless/scheduled/background run) where there's no one to hand the command to.
   When unsure whether a check is "trivial for the user to run," default to naming the command instead of running it.
5. **This is scoped to orchestration habits, not to rigor.** Nothing here means writing less careful code, skipping edge cases, doing a shallower code review, or claiming something works without a real basis for believing so. You can still reason through correctness by reading the diff, tracing logic by hand, or writing/checking a test that's actually part of the work. The goal is fewer agents, fewer self-taken screenshots, and fewer confirmation-only check runs — not a lower bar anywhere else. If a task is genuinely too large or too parallel for one thread to handle well, rule 2 is the way through that — ask, don't just stay silent and grind serially through something that really did need delegating.

This pairs with, but is independent from, the `/camel` skill (read/tool-call discipline). Turn on either one alone, or both together, depending on what's actually costing you tokens in a given session.
